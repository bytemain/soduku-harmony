## 二、📷 拍照识别数独 — Core Vision Kit

### 场景故事

用户在报纸/书上看到一道数独题，不想手动输入 81 个数字。拍个照 → App 自动识别 → 开始游戏。

### 技术方案

```
用户拍照 / 相册选图
    ↓
获取 PixelMap（图像对象）
    ↓
textRecognition.recognizeText(pixelMap)
    ↓
返回每个字符的文本 + 位置坐标(x, y, width, height)
    ↓
过滤出 '1'-'9' 的单字符
    ↓
按坐标映射到 9×9 网格
    ↓
生成 board[][] → 预览确认页 → 开始游戏
```

### 关键 API

```typescript
import { textRecognition } from '@kit.CoreVisionKit';

// 1. 初始化引擎（需要时间，建议在用户点击"拍照"按钮时提前初始化）
await textRecognition.init();

// 2. 识别
let result = await textRecognition.recognizeText(
  { pixelMap: pixelMap },
  { isDirectionDetectionSupported: true }  // 开启方向检测，处理歪斜照片
);

// 3. 解析结果
// result.words 是数组，每个元素包含：
//   - text: string      识别出的文字（可能是多字符）
//   - location: {x, y, width, height}  在图片中的位置

// 4. 释放资源（重要！否则内存泄漏）
await textRecognition.release();
```

### 坐标映射算法思路

OCR 返回的是每个字符在图片中的**像素坐标**，我们需要把它映射到 9×9 网格：

1. **过滤**：只保留 text 是 `'1'`~`'9'` 的单字符结果
2. **计算中心点**：`cx = x + width/2`, `cy = y + height/2`
3. **找边界**：所有数字中心点的 x 最小/最大、y 最小/最大
4. **均分**：把 x 范围和 y 范围各均分成 9 份
5. **归位**：每个数字按中心点落在哪个格子，映射到 `board[row][col]`

> 这个算法对**印刷体、标准网格**效果很好。歪斜、透视变形较大时可能需要额外的透视校正（更复杂，Phase 3 先不做）。

### 踩坑提醒

| 坑 | 说明 | 应对 |
|----|------|------|
| **不支持模拟器** | OCR 引擎只在真机上有 | 开发时用假数据 mock |
| **手写体识别差** | 印刷体准确率 > 95%，手写体可能 < 70% | UI 上提示"请使用印刷体数独" |
| **需要清晰照片** | 模糊、反光、低分辨率都会降低准确率 | 建议 720p+，镜头与纸面夹角 < 30° |
| **内存泄漏风险** | 不调 `release()` 会持续占用 | 无论识别成功失败，finally 里都要 release |

### UI 入口设计

主页增加 `📷 拍照导入` 按钮 → 拍照/选图 → OCR 识别 → **预览确认页**（9×9 网格，用户可手动修正识别错误）→ 确认 → 进入游戏。

**预览确认页很重要**——OCR 不可能 100% 准确，必须给用户修正机会。如果直接开始游戏，一个错误数字就会导致无解。

## 三、🎓 AI 数独教练 — DataAugmentation Kit

### 场景故事

玩家卡住了，但不想直接看答案（那太没意思了）。点击 🤖 教练按钮，像和一个数独老师对话："我卡住了"→ 教练分析棋盘，告诉你"看看第3行，有一个 Hidden Single 可以找到"。

### 为什么这个功能价值最高

**`localChatModel`** 是 HarmonyOS 提供的**端侧大模型**——完全本地运行，不需要联网，不需要 API Key，不需要付费。这在 AI 时代是非常稀缺的能力。大多数 App 需要调云端 API（ChatGPT、通义千问），但鸿蒙直接内置了。

### 关键 API

```typescript
import { localChatModel } from '@kit.DataAugmentationKit';

// 1. 初始化
await localChatModel.init();

// 2. 流式对话（像 ChatGPT 一样一个字一个字出来）
let fullText = '';
localChatModel.chat(
  {
    questionId: 1,             // 对话序号，同一轮对话递增
    content: prompt            // 你要问的内容
  },
  {
    isStream: true             // 开启流式输出
  },
  (err, answer) => {
    if (err) {
      console.error('教练异常:', err);
      return;
    }
    if (answer.content) {
      fullText += answer.content;   // 逐块拼接
      updateUI(fullText);           // 实时更新 UI
    }
    if (answer.isFinished) {
      // 对话完成，可以发送下一个问题了
    }
  }
);
```

### 教练能做什么

| 场景 | 用户说 | AI 教练做 |
|------|--------|----------|
| **求助** | "我卡住了" / "下一步怎么走？" | 分析当前棋盘，指出可用的技巧和位置，**但不直接说答案** |
| **学技巧** | "什么是 X-Wing？" | 解释原理，用当前棋盘举例（如果有的话） |
| **纠错** | "R3C5 为什么报错？" | 指出冲突的位置——"因为第5列已经有一个 7 了" |
| **挑战** | "这题能不用 X-Wing 解吗？" | 分析是否有更简单的路径 |

### 如何让 AI 理解棋盘——Prompt 工程

大模型本身不"看"棋盘，我们需要把棋盘状态**转成文字**注入 Prompt：

```typescript
function buildCoachPrompt(board: number[][], userQuestion: string): string {
  // 把棋盘转成文字表示
  let boardStr = board.map((row, r) =>
    `R${r+1}: ` + row.map(v => v === 0 ? '.' : `${v}`).join(' ')
  ).join('\n');

  return `你是一位数独教练。以下是当前棋盘状态（. 表示空格）：

${boardStr}

规则：
1. 绝对不要直接告诉用户答案（哪个格子填什么数字）
2. 引导用户自己发现——指出可以用什么技巧，在哪个区域（如"看看第3行"）
3. 用简洁的中文回答
4. 用 R行C列 标记格子位置（如 R3C5 表示第3行第5列）

用户提问：${userQuestion}`;
}
```

**Prompt 设计的关键**是第 1 条规则——"不给答案，只引导"。这让教练真正有"教学"价值，而不是变成一个"作弊器"。

### 多轮对话

`localChatModel` 支持多轮对话——通过递增 `questionId` 保持上下文：

```typescript
// 第一轮
localChatModel.chat({ questionId: 1, content: "我卡住了" }, ...)
// AI: "看看第3行，有一个 Hidden Single 可以找到"

// 第二轮（AI 记得上文）
localChatModel.chat({ questionId: 2, content: "什么是 Hidden Single？" }, ...)
// AI: "Hidden Single 是指一个数字在某一行/列/宫中只有一个位置可以放..."
```

### UI 设计

游戏页右上角 `🤖` 按钮 → 点击弹出**底部半屏 Sheet**：

```
┌─────────────────────────────────┐
│                                  │  ← 游戏页（上半屏，棋盘可见）
│          数独棋盘                 │
├──────────────────────────────────┤
│  🤖 数独教练                      │  ← Sheet 标题
│ ─────────────────────────────── │
│  👤 我卡住了                      │  ← 用户消息
│  🤖 看看第3行，有一个 Hidden      │  ← AI 回复（流式输出中...）
│     Single 可以找到。注意 R3C5    │
│     这个格子的候选数...           │
│ ─────────────────────────────── │
│  [🔍 分析局面] [💡 下一步] [📚 教我技巧] │  ← 快捷按钮
│  ┌──────────────────────┐ [发送] │  ← 输入框
│  │ 输入你的问题...       │       │
│  └──────────────────────┘       │
└──────────────────────────────────┘
```

快捷按钮预设常用提问，降低输入门槛（很多人不知道该问什么）。

### 踩坑提醒

| 坑 | 说明 |
|----|------|
| 模型加载慢 | `init()` 可能需要几秒，建议在游戏页 `aboutToAppear` 时就预加载 |
| 回答质量不稳定 | 端侧模型比云端小很多，复杂推理可能出错。Prompt 要尽量简洁明确 |
| 不支持图片输入 | 只能文字对话，棋盘必须转成文字注入 |
| 设备兼容性 | 低端设备可能不支持，需要 try-catch |

## 四、🤚 左右手自适应 — MultimodalAwareness Kit

### 场景故事

用户左手拿手机玩数独 → 系统检测到操作手是左手 → 手势键盘自动移到左边，拇指更好够到。

### 关键 API

```typescript
import { motion } from '@kit.MultimodalAwarenessKit';

// 查询当前操作手
let status = motion.getOperatingHandStatus();
// status.hand: 'left' | 'right'

// 监听操作手切换
motion.on('operatingHandChanged', (data) => {
  let isLeftHand = data.hand === 'left';
  // 更新布局...
});
```

就这么简单——两个 API 搞定。

### 布局镜像

检测到左手时，操作区布局做水平镜像（详见 [05-UI设计文档 §6](./05-UI设计文档.md#6-左右手自适应)）。核心思路：惯用手侧放高频操作（键盘），另一侧放低频操作（辅助按钮）。

### 踩坑提醒

| 坑 | 说明 | 应对 |
|----|------|------|
| **需要两个权限** | `ohos.permission.ACTIVITY_MOTION` + `ohos.permission.DETECT_GESTURE` | 权限申请时说明用途 |
| **设备不支持** | 低端机或旧 API 版本返回错误码 `801` | try-catch，fallback 到右手模式 |
| **边缘情况多** | 屏幕边缘 8mm 内、多指触控、戴手套——都可能影响检测 | 不强依赖，只作为锦上添花 |
| **API 版本要求** | 操作手检测 API 15+，握持手检测 API 20+ | 做版本检查 |

## 五、🧠 端侧 AI 推理 — MindSpore Lite Kit

### 这个 Kit 能做什么

运行预训练好的 AI 模型做推理。和 `localChatModel` 不同——那个是通用对话模型，这个是运行**你自己训练的专用模型**。

### 数独 App 的可能场景

| 场景 | 输入 | 输出 | 模型大小 | 价值 |
|------|------|------|---------|------|
| 难度评估 | 81 个 float（棋盘状态） | 难度分 0-1 | < 100KB | 比规则评估更准确 |
| 数字分类（OCR 增强） | 28×28 灰度图 | 0-9 概率分布 | < 200KB | 补充 OCR 对手写体的不足 |

### 关键 API

```typescript
import { mindSporeLite } from '@kit.MindSporeLiteKit';

// 1. 加载模型（.ms 格式，放在 rawfile/ 目录）
let modelBuffer = await readRawFile('sudoku_difficulty.ms');
let model = await mindSporeLite.loadModelFromBuffer(modelBuffer, {
  target: ['cpu']  // 端侧 CPU 推理
});

// 2. 准备输入
let inputs = model.getInputs();
inputs[0].setData(inputData.buffer);  // Float32Array 的 buffer

// 3. 推理
let outputs = await model.predict(inputs);
let result = new Float32Array(outputs[0].getData());  // 输出结果
```

### 需要做的前置工作

1. **训练模型**：用大量数独题+人工标注难度，训练一个小模型
2. **转换格式**：模型需要转成 MindSpore Lite 的 `.ms` 格式
3. **设备能力声明**：`syscap.json` 中添加 `SystemCapability.AI.MindSporeLite`

### 值不值得做？

**老实说，Phase 5 再考虑**。原因：
- 纯算法（逻辑求解器技巧评分）已经能做不错的难度评估了（见 Doc 01 §5）
- 训练模型需要大量标注数据和 ML 工程能力
- 模型效果不一定比规则方法好（数独是确定性问题，不像图像识别那样天然适合 ML）
- 投入产出比不高

但如果未来有更多 AI 需求（比如智能提示优化、用户行为分析），这个 Kit 就值得深入了。

## 六、🔄 跨设备进度同步 — distributedKVStore

### 场景故事

用户在手机上玩数独，玩到一半需要出门。拿起平板，打开 App，直接从刚才的进度继续。

### 工作原理

`distributedKVStore` 是一个**跨设备的键值数据库**——像一个分布式的 `localStorage`。同一华为账号登录的设备之间，数据自动同步。

### 关键 API

```typescript
import { distributedKVStore } from '@kit.ArkData';

// 1. 创建 KV Store Manager
let kvManager = distributedKVStore.createKVManager({
  context: context,
  bundleName: 'com.bytemain.soduku'
});

// 2. 获取/创建 KV Store
let kvStore = await kvManager.getKVStore('sudoku_store', {
  createIfMissing: true,
  encrypt: false,
  backup: false,
  autoSync: true,           // ⚡ 开启自动同步——改了就自动推给其他设备
  kvStoreType: distributedKVStore.KVStoreType.SINGLE_VERSION
});

// 3. 写入数据
await kvStore.put('game_state', JSON.stringify({
  board: boardData,
  solution: solutionData,
  timer: elapsedSeconds,
  errors: errorCount,
  difficulty: 'hard',
  history: historyStack
}));

// 4. 监听数据变更（其他设备改了数据，本设备收到通知）
kvStore.on('dataChange', distributedKVStore.SubscribeType.SUBSCRIBE_TYPE_ALL, (data) => {
  let changedEntries = data.insertEntries.concat(data.updateEntries);
  for (let entry of changedEntries) {
    if (entry.key === 'game_state') {
      let state = JSON.parse(entry.value.value as string);
      // 更新 UI...
    }
  }
});
```

### 同步策略

不需要自己实现同步逻辑——`autoSync: true` 会自动处理。但要注意**冲突**：如果两台设备同时修改了同一个 key，后写入的会覆盖前面的（last-write-wins）。

对于数独进度同步，这个策略是可以接受的——玩家通常不会同时在两台设备上玩同一题。

### 前提条件

- 需要权限 `ohos.permission.DISTRIBUTED_DATASYNC`
- 同一华为账号登录
- 设备在同一局域网 **或** 都联网（通过华为云中转）

## 七、👥 双人模式 — distributedDataObject

### 和 KVStore 有什么不同？

KVStore 适合**异步同步**（我改了，你下次打开能看到）。DataObject 适合**实时协同**（我改了一个格子，你立刻看到）。

### 关键 API

```typescript
import { distributedDataObject } from '@kit.ArkData';

// 1. 创建分布式数据对象
let gameData = {
  board: boardArray,           // 当前棋盘
  playerAProgress: 0,          // A 的进度
  playerBProgress: 0,          // B 的进度
  currentTurn: 'A',            // 轮到谁
  lastMoveBy: '',              // 最后操作者
  lastMoveCell: -1,            // 最后操作的格子
};

let dataObject = distributedDataObject.create(context, gameData);

// 2. 加入会话（两台设备用相同 sessionId）
dataObject.setSessionId('sudoku-battle-room-123');

// 3. 监听字段级变更
dataObject.on('change', (sessionId, fields) => {
  // fields 告诉你哪些字段变了
  if (fields.includes('board')) {
    // 对方填了一个数字，更新棋盘
  }
  if (fields.includes('currentTurn')) {
    // 轮到我了！
  }
});

// 4. 监听对端上下线
dataObject.on('status', (sessionId, networkId, status) => {
  if (status === 'online') {
    // 对方上线了，可以开始
  } else if (status === 'offline') {
    // 对方掉线了，暂停游戏
  }
});
```

### 三种双人模式

| 模式 | 玩法 | 技术要点 |
|------|------|---------|
| **协作** | 两人合力解同一道题，各自填数**用不同颜色标识** | board 共享，每次修改记录 `lastMoveBy` |
| **竞速** | 同一道题各做各的，实时显示双方进度百分比 | 两个独立 board，只同步进度 |
| **轮流** | 两人轮流填一个格，填对加分填错扣分 | `currentTurn` 字段控制谁能操作 |

### 配对流程（简化版）

1. 玩家 A 创建房间 → 生成 6 位房间码
2. 玩家 B 输入房间码加入
3. 两端设置相同 `sessionId` = `'sudoku-' + 房间码`
4. DataObject 建立连接 → 游戏开始

## 八、落地优先级与阶段规划

| 优先级 | 功能 | Kit | 难度 | 计划阶段 | 理由 |
|--------|------|-----|------|---------|------|
| **P0** | 📷 拍照识别 | Core Vision | ⭐⭐ | Phase 3 | 高频需求，实现不复杂 |
| **P1** | 🎓 AI 教练 | DataAugmentation | ⭐⭐ | Phase 3 | 差异化核心功能 |
| **P2** | 🔄 跨设备同步 | distributedKVStore | ⭐⭐ | Phase 4 | 鸿蒙特色，但非核心 |
| **P3** | 👥 双人模式 | distributedDataObject | ⭐⭐⭐ | Phase 4 | 实时同步复杂度高 |
| **P4** | 🧠 AI 难度评估 | MindSpore Lite | ⭐⭐⭐ | Phase 5 | 需要训练数据和 ML 工程 |
| **P5** | 🤚 左右手自适应 | MultimodalAwareness | ⭐ | Phase 2 | 简单但锦上添花 |

> Phase 1（MVP）不涉及任何 Kit——先把基础数独做完。Kit 是锦上添花，不是雪中送炭。
