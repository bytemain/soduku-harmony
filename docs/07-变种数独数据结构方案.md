# 🧩 变种数独数据结构方案

> 分析接入 Sudoku-X、Jigsaw、Killer、可变尺寸等变种数独的数据结构设计。
>
> 参考：[Simon Tatham Solo](https://www.chiark.greenend.org.uk/~sgtatham/puzzles/doc/solo.html)、Doc 03 Phase 5

## 一、现状分析

### 1.1 当前硬编码假设

| 文件 | 硬编码点 | 影响 |
|------|---------|------|
| `GameConstants.ets` | `GRID_SIZE=9`, `BOX_SIZE=3`, `TOTAL_CELLS=81` | 全局常量，所有模块依赖 |
| `Topology.ets` | 固定 27 个 units（9行+9列+9宫），模块加载时预算 | 不支持对角线/不规则宫 |
| `CellData.ets` | `box` getter 硬编码 `row/3*3 + col/3` | 仅适用 3×3 标准宫 |
| `CellData.ets` | `createEmptyBoard()` 固定 81 格 | 不支持其他尺寸 |
| `GameState.ets` | `selectedIndex >= 81`、`solution` 固定 81 长度 | 绑定 9×9 |
| `SudokuSolver.ets` | 导入 `PEERS`, `UNITS_OF` 全局常量 | 只能用标准拓扑 |
| `SudokuValidator.ets` | 导入 `PEERS`、循环 `TOTAL_CELLS` | 只能用标准拓扑 |
| `SudokuGenerator.ets` | `GRID_SIZE` 控制循环边界 | 仅 9×9 |
| `NoteState` | 每数字 2 bit，共 18 bit（9 个数字） | 12×12 需要 24 bit（仍在 32 bit 内） |

### 1.2 已有的良好抽象

| 设计 | 说明 | 复用性 |
|------|------|--------|
| `CellFlags` 位标志 | 通用状态管理，与网格尺寸无关 | ✅ 直接复用 |
| `NoteState` 2-bit 位图 | 每数字 2 bit，`number` 32 bit 最多支持 16 个数字 | ✅ 支持到 16×16 |
| `Topology` 的 units/peers 模型 | 约束关系完全由 units 数组定义 | ✅ 只需参数化构建 |
| `SudokuSolver` 位图候选集 | 当前 9 bit，扩展到 12/16 bit 即可 | ✅ 改 `ALL_CANDIDATES` |
| `SeededRandom` + Seed 系统 | 与网格尺寸/变体无关 | ✅ 直接复用 |
| `Symmetry` 对称模式 | 位置计算需参数化 `GRID_SIZE` | ⚠️ 小改 |

## 二、目标变种

### 2.1 Sudoku-X（对角线数独）

**规则变化**：在标准 9×9 基础上，两条主对角线也必须包含 1-9。

**影响范围**：

| 层面 | 变化 | 复杂度 |
|------|------|--------|
| 拓扑 | 新增 2 个 units（主对角线、副对角线），对角格的 peers 增多 | 低 |
| 求解 | Solver 无需改动（已依赖 units/peers 抽象） | 无 |
| 生成 | 终盘生成需在对角线上也满足约束 | 低 |
| UI | 对角线格子加底色/阴影标识 | 低 |

### 2.2 Jigsaw（不规则宫数独）

**规则变化**：9 个区域不再是 3×3 方块，而是任意形状的 9 连通区域。

**影响范围**：

| 层面 | 变化 | 复杂度 |
|------|------|--------|
| 拓扑 | 9 个 box units 由 `regionMap` 定义，非固定 3×3 | 中 |
| 数据模型 | 每个 Cell 需知道所属 region ID | 低 |
| 生成 | 需生成合法的不规则区域分割 + 基于区域的终盘填充 | 高 |
| UI | 区域边界渲染（相邻格不同 region 时画粗线） | 中 |

### 2.3 Killer（杀手数独）

**规则变化**：格子分组为"笼子"（cage），每个笼子有目标和数，笼子内数字不能重复。

**影响范围**：

| 层面 | 变化 | 复杂度 |
|------|------|--------|
| 数据模型 | 新增 `KillerCage` 结构（格子列表 + 目标和） | 中 |
| 拓扑 | 每个笼子形成新 unit（笼子内不能重复） | 低 |
| 求解 | 需加和约束剪枝（笼子总和 = target） | 高 |
| 生成 | 生成合法笼子分区 + 计算各笼目标和 | 高 |
| UI | 笼子虚线边界 + 左上角显示目标和数字 | 中 |

### 2.4 可变尺寸（4×4 / 6×6 / 12×12 / 16×16）

**规则变化**：网格尺寸可变，宫格为 `boxRows × boxCols`。

**影响范围**：

| 层面 | 变化 | 复杂度 |
|------|------|--------|
| 常量 | `GRID_SIZE`, `BOX_SIZE`, `TOTAL_CELLS`, `DIGITS` 全部参数化 | 中 |
| 数据模型 | Cell 数量动态，value 范围动态（12×12 用 1-12） | 中 |
| 拓扑 | `buildAllUnits()` 参数化 `gridSize`, `boxRows`, `boxCols` | 低 |
| 笔记 | 12 个数字 × 2 bit = 24 bit（`number` 够用） | 低 |
| 求解 | `ALL_CANDIDATES` 动态计算 | 低 |
| UI | 键盘布局需适配（12 键、16 键） | 中 |

## 三、数据结构设计

### 3.1 VariantType 枚举

```typescript
export enum VariantType {
  /** 标准 9×9 */
  STANDARD = 'standard',
  /** 对角线数独（Sudoku-X） */
  DIAGONAL = 'diagonal',
  /** 不规则宫数独（Jigsaw） */
  JIGSAW = 'jigsaw',
  /** 杀手数独（Killer） */
  KILLER = 'killer',
}
```

**设计决策**：用 string enum 而非 number，便于序列化/日志阅读（与现有 `Symmetry` 风格一致）。

### 3.2 GridConfig — 棋盘几何配置

```typescript
export interface GridConfig {
  /** 棋盘边长（4/6/9/12/16） */
  gridSize: number
  /** 宫格行数（如 9×9 标准为 3） */
  boxRows: number
  /** 宫格列数（如 9×9 标准为 3） */
  boxCols: number
  /** 总格数 = gridSize² */
  totalCells: number
  /** 有效数字列表，如 [1..9] 或 [1..12] */
  digits: number[]
}
```

**预设**：

| 名称 | gridSize | boxRows | boxCols | 说明 |
|------|----------|---------|---------|------|
| `GRID_4x4` | 4 | 2 | 2 | 入门 |
| `GRID_6x6` | 6 | 2 | 3 | 简易 |
| `GRID_9x9` | 9 | 3 | 3 | 标准 |
| `GRID_12x12` | 12 | 3 | 4 | 进阶 |
| `GRID_16x16` | 16 | 4 | 4 | 挑战 |

### 3.3 RegionConfig — 区域定义（Jigsaw 用）

```typescript
export interface RegionConfig {
  /** 区域数量（通常 = gridSize） */
  regionCount: number
  /**
   * 每个格子所属的区域 ID (0-based)。
   * 长度 = totalCells，regionMap[index] = regionId
   */
  regionMap: number[]
}
```

**说明**：
- 标准数独的 `regionMap` 就是每个格子的 box 编号（`Math.floor(r/3)*3 + Math.floor(c/3)`）。
- Jigsaw 数独由生成算法产出一个不规则的 `regionMap`。
- `regionMap` 同时驱动：拓扑构建（units 生成）、UI 渲染（边界判定）。

### 3.4 KillerCage — 笼子（Killer 用）

```typescript
export interface KillerCage {
  /** 笼子内格子的一维索引列表 */
  cells: number[]
  /** 笼子的目标和 */
  sum: number
}
```

**约束**：
- 一个格子只属于一个笼子。
- 笼子内数字不能重复（自动成为一个 unit）。
- `sum` 约束在求解/验证时额外检查。

### 3.5 VariantConfig — 变种完整配置

```typescript
export interface VariantConfig {
  /** 变种类型 */
  type: VariantType
  /** 棋盘几何 */
  grid: GridConfig
  /** 是否启用对角线约束 */
  hasDiagonals: boolean
  /** 自定义区域（Jigsaw 用，null 表示用标准 box 分区） */
  regions: RegionConfig | null
  /** 笼子列表（Killer 用，null 表示无笼子） */
  cages: KillerCage[] | null
}
```

**设计理念**：
- **组合优于继承**：`hasDiagonals`, `regions`, `cages` 是独立开关，可以自由组合（如 "Killer + 对角线"）。
- **标准数独是特例**：`type=STANDARD, hasDiagonals=false, regions=null, cages=null`，零额外开销。
- **配置驱动拓扑**：Topology 模块根据 `VariantConfig` 动态构建 units/peers，Solver/Validator 不需要知道变体类型。

### 3.6 预设配置工厂

```typescript
/** 创建标准 9×9 配置 */
function createStandardConfig(): VariantConfig

/** 创建 Sudoku-X 配置 */
function createDiagonalConfig(): VariantConfig

/** 创建 Jigsaw 配置（需传入 regionMap） */
function createJigsawConfig(regionMap: number[]): VariantConfig

/** 创建 Killer 配置（需传入 cages） */
function createKillerConfig(cages: KillerCage[]): VariantConfig

/** 创建任意尺寸标准配置 */
function createSizedConfig(gridSize: number, boxRows: number, boxCols: number): VariantConfig
```

## 四、拓扑构建方案

### 4.1 VariantTopology — 动态拓扑

核心思路：**从 `VariantConfig` 计算出 units 和 peers，接口不变，只是构建方式参数化。**

```
VariantConfig
    │
    ▼
buildVariantUnits(config)
    │
    ├─ 行 units（gridSize 个）
    ├─ 列 units（gridSize 个）
    ├─ 区域 units（标准 box 或自定义 region）
    ├─ 对角线 units（0 或 2 个）
    └─ 笼子 units（0 或 N 个）
    │
    ▼
buildVariantPeers(units, totalCells)
    │
    ▼
VariantTopology { allUnits, unitsOf, peers }
```

**与现有 `Topology.ets` 的关系**：

- 现有的 `ALL_UNITS`, `UNITS_OF`, `PEERS` 保持不变（零改动，标准 9×9 走老路）。
- 新增 `VariantTopology` 类/接口，仅在变种模式下使用。
- Solver/Validator 新增接受 `peers`/`unitsOf` 参数的重载，默认值为标准拓扑。

### 4.2 行/列 units 构建

```typescript
// gridSize 个行 units
for (let row = 0; row < gridSize; row++) {
  unit = [row * gridSize + 0, row * gridSize + 1, ..., row * gridSize + (gridSize-1)]
}

// gridSize 个列 units
for (let col = 0; col < gridSize; col++) {
  unit = [0 * gridSize + col, 1 * gridSize + col, ..., (gridSize-1) * gridSize + col]
}
```

### 4.3 区域 units 构建

**标准模式**（`regions === null`）：

```typescript
for (let br = 0; br < gridSize; br += boxRows) {
  for (let bc = 0; bc < gridSize; bc += boxCols) {
    unit = 遍历 [br..br+boxRows) × [bc..bc+boxCols) 的所有格子索引
  }
}
```

**Jigsaw 模式**（`regions !== null`）：

```typescript
// 按 regionId 分组
for (let regionId = 0; regionId < regions.regionCount; regionId++) {
  unit = regionMap 中所有值为 regionId 的格子索引
}
```

### 4.4 对角线 units 构建

```typescript
if (hasDiagonals) {
  // 主对角线：(0,0), (1,1), ..., (gridSize-1, gridSize-1)
  mainDiag = [0, gridSize+1, 2*(gridSize+1), ...]

  // 副对角线：(0, gridSize-1), (1, gridSize-2), ...
  antiDiag = [gridSize-1, 2*(gridSize-1), ...]

  units.push(mainDiag, antiDiag)
}
```

### 4.5 笼子 units 构建

```typescript
if (cages !== null) {
  for (const cage of cages) {
    units.push(cage.cells)  // 笼子内不重复 → 形成 unit
  }
}
```

## 五、各模块改造影响

### 5.1 改造影响矩阵

| 模块 | 改动类型 | 改动量 | 说明 |
|------|---------|--------|------|
| `GameConstants.ets` | 不改 | 无 | 现有常量保留作为默认值 |
| `CellData.ets` | 新增字段 | 小 | 加 `regionId` 可选字段；`createEmptyBoard` 接受 `totalCells` 参数 |
| `GameState.ets` | 新增字段 | 小 | 加 `variantConfig: VariantConfig`；参数化 `totalCells` |
| `Topology.ets` | 不改 | 无 | 现有导出保持向后兼容 |
| **新增 `VariantTopology.ets`** | 新模块 | 中 | 根据 VariantConfig 构建动态拓扑 |
| **新增 `VariantConfig.ets`** | 新模块 | 中 | 枚举 + 接口 + 工厂函数 |
| `SudokuSolver.ets` | 新增重载 | 小 | 增加接受 topology 参数的函数 |
| `SudokuValidator.ets` | 新增重载 | 小 | 增加接受 topology 参数的函数 |
| `SudokuGenerator.ets` | 未来扩展 | 中 | 各变种需定制生成逻辑（Phase 5 具体实现时处理） |
| `SudokuBoard.ets` | 未来扩展 | 中 | 边界渲染逻辑（不规则区域/笼子边框） |
| `SudokuCell.ets` | 未来扩展 | 小 | 对角线背景色、笼子和数字显示 |

### 5.2 向后兼容策略

**核心原则：标准 9×9 数独零改动，变种是增量扩展。**

```
标准模式调用链（不变）：
  Topology.PEERS → SudokuValidator.validateAllCells(cells)

变种模式调用链（新增）：
  VariantConfig → VariantTopology.build(config)
    → { peers, unitsOf } → SudokuValidator.validateAllCells(cells, solution, peers)
```

- 所有现有函数保持签名不变。
- 新增接受 topology 参数的函数变体（如 `validateAllCellsWithTopology`）。
- `GameState` 新增 `variantConfig` 字段，默认值为 `STANDARD_CONFIG`。

export class CellData {

```typescript
/** 当前格的区域编号 */
get box(): number {
  // 优先返回 regionId（Jigsaw 模式下由外部设置）
  // 标准模式下 regionId 默认等于计算值
  return this.regionId
}
```

## 八、序列化 / Seed 系统扩展

### 8.1 Seed 编码

现有格式：`difficulty:symmetry:seed`（如 `3:rot180:123456789`）

扩展格式：`variant:gridSize:difficulty:symmetry:seed`

```typescript
// 标准模式（向后兼容）
"3:rot180:123456789"

// Sudoku-X
"diagonal:9:3:rot180:123456789"

// Jigsaw（region 信息太长，不编入 seed，由 seed 确定性生成）
"jigsaw:9:3:none:123456789"

// Killer
"killer:9:3:none:123456789"

// 4×4
"standard:4:1:rot180:123456789"
```

### 8.2 存档扩展

`GameStorage` 存档增加 `variantConfig` 字段，加载时根据 type 重建拓扑。

## 十、总结

| 设计要点 | 方案 |
|---------|------|
| 变种抽象 | `VariantConfig` 组合式配置（type + grid + diagonals + regions + cages） |
| 拓扑抽象 | `VariantTopology` 从配置动态构建 units/peers，Solver/Validator 依赖接口不变 |
| 向后兼容 | 现有 `Topology.ets` 常量和函数签名零改动，标准模式无性能损失 |
| 扩展性 | 新增变种只需：① 扩展 `VariantType` ② 实现生成逻辑 ③ 适配 UI |
| CellData 扩展 | 新增 `regionId` + `cageIndex`，存量代码不受影响 |
| Seed 兼容 | 新格式向后兼容旧格式，标准模式 seed 不变 |
