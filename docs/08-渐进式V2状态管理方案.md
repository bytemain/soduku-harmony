# 渐进式 V2 状态管理方案

> 说明：本方案已同时参考官方文档：
>
> - `docs/harmonyos-guides/arkts-mvvm.md`（MVVM模式（V1））
> - `docs/harmonyos-guides/arkts-mvvm-v2.md`（MVVM模式（V2））
>
> 结论很明确：**V1 和 V2 都有自己的完整 MVVM 落地方式，不能把 V2 ViewModel 硬塞进 V1 页面里当“半桥接状态层”。**

## 0. 先说结论

结合两份官方文档，本项目当前最合理的策略是：

1. **现有 `GamePage.ets` 继续保持 V1 主线稳定**。
2. **V2 以新子树 / 新页面的形式落地，而不是反向污染 V1 页面**。
3. **不要再做“V1 页面 + V2 store mirror”这类中间态**，因为它既不是官方 V1 MVVM，也不是官方 V2 MVVM。

换句话说：

- 要么你在 V1 里老老实实做 V1 MVVM。
- 要么你在 V2 里老老实实做 `@ComponentV2 + @Local/@Param/@Event + @ObservedV2`。
- 中间那层“假 V2”桥接，官方文档没有把它当推荐方案。

## 0.1 官方 V1 / V2 的边界

### 官方 MVVM V1 的典型形态

从 `arkts-mvvm.md` 看，V1 的重点是：

- `@Component`
- `@State`
- `@Link`
- `@Prop`
- `@Observed`
- `@ObjectLink`

也就是说，**V1 的 ViewModel / 状态对象，是围绕 V1 状态装饰器和 V1 组件树运转的**。

### 官方 MVVM V2 的典型形态

从 `arkts-mvvm-v2.md` 看，V2 的重点是：

- `@ComponentV2`
- `@Local`
- `@Param`
- `@Event`
- `@ObservedV2`
- `@Trace`
- 需要时再配 `@Computed`、`@Monitor`
- 跨页面共享时用 `AppStorageV2`
- 持久化时用 `PersistenceV2`

也就是说，**V2 的 ViewModel 是给 V2 组件树消费的，不是给 V1 `@State` 页面临时过桥的**。

## 0.2 本项目为什么当前不直接“整体切 V2”

当前仓库的核心现实是：

- `GamePage.ets` 仍是大体量 V1 页面
- `SudokuBoard` 链路依赖 V1 的 `@ObjectLink`
- 现有交互、动画、Worker、历史栈都已经和 V1 页面结构深度耦合

因此，如果现在强行整体切 `@ComponentV2`，风险会很高：

- 棋盘刷新链路容易被改坏
- 手势 / 多选 / 历史回退的行为容易回归
- Worker 生命周期和页面生命周期更难一次性理顺

所以当前阶段正确做法不是“把 V2 塞进 V1 页面”，而是：

- **V1 主页面继续稳定运行**
- **V2 以独立边界逐步引入**

## 1. 这次为什么先回滚单体 `GamePageStore`

上一版把 `GamePage` 的几十个字段机械搬进了一个 `GamePageStore`，但有两个根本问题：

1. **只是换了放置位置，没有改善归属关系**：加载态、UI 态、交互态、运行时句柄、历史栈都被塞进一个对象里，领域边界依旧混乱。
2. **和 Harmony V2 目标不一致**：页面仍然是 V1 组件树，强行套一个 V1 单体 store，既不能发挥 `@ObservedV2` 的优势，也会让后续迁移路径更模糊。

因此这次先做两件事：

- 把 `GamePage.ets` 恢复成直接持有状态，保证当前主线稳定。
- 单独补一套**不直接接入当前 V1 页面运行时**的 V2 分域骨架，作为后续 `ComponentV2 + ViewModel` 迁移目标。

## 1.1 参考官方文档后的关键修正

根据 `docs/harmonyos-guides/arkts-mvvm-v2.md`，V2 的推荐路径不是“把 V2 store 硬塞给 V1 页面”，而是：

- 页面 / 视图使用 `@ComponentV2`
- 页面内部用 `@Local` 持有 `@ObservedV2` ViewModel
- 子组件通过 `@Param` 接收 ViewModel / 数据
- 子组件通过 `@Event` 向父组件回传交互
- 复杂嵌套数据使用 `@ObservedV2 + @Trace`
- 跨页面共享 UI 状态时再考虑 `AppStorageV2`
- 需要持久化时再考虑 `PersistenceV2`

这意味着：

- `GamePage.ets` 当前还是 V1 `@Component`，**不能算真正进入官方 V2 MVVM**。
- `GamePage.ets` 当前也不该伪装成 V2 的宿主，因为它没有变成官方文档里的 `@ComponentV2` 页面。
- 之前那种“V1 页面持有 V2 对象，再手动同步到 `@State`”的写法，只是实验性桥接，**不属于官方推荐范式**。
- 因此现在已把这层桥接撤回，避免继续偏离官方模式。

## 2. 分域原则

目标不是“所有状态都进一个 store”，而是“每种状态回到它真正的 owner”。

### 2.1 Session Domain

负责一局游戏的会话事实：

- `difficulty`
- `symmetry`
- `seedId`
- `importedPuzzle`
- `elapsedSeconds`
- `errorCount`
- `isCompleted`
- `isPaused`

对应文件：`entry/src/main/ets/stores/v2/GameSessionDomainV2.ets`

### 2.2 UI Domain

负责纯显示层状态：

- `showSettings`
- `showPauseOverlay`
- `showGestureGuide`
- `showCoordinates`
- `showAutoNotes`
- `boardScale`
- `boardOpacity`
- `completionScale`
- `numPadHeight`

对应文件：`entry/src/main/ets/stores/v2/GameUiDomainV2.ets`

### 2.3 Loading Domain

负责初始化 / 导入 / 生成过程中的加载反馈：

- `isLoading`
- `title`
- `message`
- `progress`

对应文件：`entry/src/main/ets/stores/v2/GameLoadingDomainV2.ets`

### 2.4 Interaction Domain

负责当前交互过程中的短生命周期状态：

- `selectedIndex`
- `gestureDirection`
- `hintFocusIndex`
- `isHintLoading`
- `isMultiSelectMode`
- `multiSelectIndices`

对应文件：`entry/src/main/ets/stores/v2/GameInteractionDomainV2.ets`

### 2.5 History Domain

负责能直接驱动 UI 的历史派生状态：

- `canUndo`
- `canRedo`
- `remainingCounts`

对应文件：`entry/src/main/ets/stores/v2/GameHistoryDomainV2.ets`

### 2.6 Runtime Context

这部分**不应该强行做成响应式 V2 store**，因为它们更像运行时句柄：

- `timerId`
- `history`（当前仍是 V1 `GameHistory`）
- 鼠标拖拽过程变量

对应文件：`entry/src/main/ets/stores/v2/GameRuntimeContext.ets`

而 `puzzleWorker` 不属于这里：它就是 `GamePage` 的实例属性，跟页面生命周期绑定，不进入任何 store。

## 3. 为什么 `runtime` 不进 `@ObservedV2`

`worker.ThreadWorker`、定时器 id、撤销栈对象这类数据不是“声明式 UI 状态”，而是运行时控制对象。

如果把它们也塞进响应式 store，会出现：

- 响应式语义不纯
- 序列化 / 持久化边界不清晰
- 组件刷新原因难以推断

所以 V2 方案里应该区分：

- **可观察 UI 状态**：进入 `@ObservedV2 + @Trace`
- **运行时控制句柄**：保留普通 class / controller / service

## 4. 当前新增骨架

本次新增但**暂未接入页面运行时**的文件：

- `entry/src/main/ets/stores/v2/GameSessionDomainV2.ets`
- `entry/src/main/ets/stores/v2/GameUiDomainV2.ets`
- `entry/src/main/ets/stores/v2/GameLoadingDomainV2.ets`
- `entry/src/main/ets/stores/v2/GameInteractionDomainV2.ets`
- `entry/src/main/ets/stores/v2/GameHistoryDomainV2.ets`
- `entry/src/main/ets/stores/v2/GameRuntimeContext.ets`
- `entry/src/main/ets/stores/v2/GamePageStateV2.ets`

这套骨架的目的只有一个：先把**未来状态边界**和 **ViewModel 拆分方向**定清楚，而不是立即把 `GamePage` 强行改成半 V1 半 V2。

## 5. 当前代码状态（按官方文档修正后）

当前仓库状态已经调整为：

- `GamePage.ets` 保持 V1 `@Component` 直持本地状态
- `entry/src/main/ets/stores/v2/` 保留为 **未来 V2 ViewModel / domain 骨架**
- `puzzleWorker` 保持 `GamePage` 实例属性，不进入任何 store
- 不再建议在 `GamePage.ets` 内继续做 V2 mirror / bridge

这也意味着当前代码的定位很清楚：

- **运行时主线**：V1
- **未来重构方向**：V2 MVVM
- **中间过渡原则**：只做边界清理和分层准备，不做伪 V2 接线

这一步的目标不是“先桥接一点点”，而是先把方向纠正为：**V1 页面维持稳定；V2 只在真正的 `@ComponentV2` 视图树里落地。**

## 6. 推荐迁移顺序

建议按“最独立、最少副作用”的顺序推进，而不是一次性把 `GamePage` 改成 V2。

### Phase 1：先拆出真正的 V2 页面岛 / V2 子树

原因：

- 和棋盘数据耦合最少
- 基本不碰撤销栈 / Worker / 数独逻辑
- 容易验证视觉行为是否一致

建议先让：

- `GameLoadingView`
- `PauseOverlay`
- 设置面板相关逻辑

但注意：这里的“迁”不是把状态塞进 V1 页面，而是：

- 新建 `@ComponentV2` 容器或子页面
- 在该 V2 组件中使用 `@Local` 持有 `GameUiDomainV2` / `GameLoadingDomainV2`
- 通过 `@Param` 把必要数据传给更细的 V2 子组件
- 通过 `@Event` 把用户操作回传给父级

这是官方文档中 `TodoList -> ViewModel -> View` 的标准路线。

对于本项目，更具体地说，第一批最适合拆成 V2 子树的是：

- 加载态覆盖层
- 暂停态覆盖层
- 设置相关小面板

因为它们：

- 对棋盘 `@ObjectLink` 主链路干扰最小
- View / ViewModel 边界更清晰
- 最接近官方 V2 示例里“视图消费 ViewModel”的结构

### Phase 2：再迁 `Interaction`

把选中格、手势方向、多选、hint 聚焦迁到 `GameInteractionDomainV2`，但前提是这些消费方已经落在 `@ComponentV2` 子树中。

这一阶段要特别注意：

- `ArrayWrapper` → `SudokuBoard` 的 `@ObjectLink` 链路不能破坏
- 不要为了“全 V2”而直接改坏当前棋盘刷新机制

### Phase 3：收敛 `Session`

把时间、错误数、完成态、暂停态等会话状态迁到 `GameSessionDomainV2`。

这一阶段会涉及：

- 计时器更新
- 完成判定
- 存档恢复后的状态回填

如果后续需要跨页面共享某些会话 UI 偏好，再按官方文档选择 `AppStorageV2`；如果需要跨重启恢复，再选 `PersistenceV2`。

### Phase 4：最后处理 `History` 与 `Runtime`

最难的是撤销/重做和 Worker 生命周期，因为这部分既有 UI 派生状态，也有运行时控制语义。

建议做法：

- UI 只依赖 `GameHistoryDomainV2` 中的 `canUndo/canRedo/remainingCounts`
- 实际撤销栈仍由 `GameRuntimeContext.history` 管理
- 等 `GameHistory` 本身有必要演进时，再单独设计

## 7. `AppStorageV2` / `PersistenceV2` 的使用建议

不是所有 domain 都应该进全局存储。

### 适合考虑 `AppStorageV2`

- 跨页面共享且不需要持久化的 UI 配置
- 例如全局偏好、主题辅助状态

### 适合考虑 `PersistenceV2`

- 需要跨启动恢复的用户偏好
- 例如 `showCoordinates`、`showAutoNotes` 这种偏好型开关

### 不适合放进去的内容

- `GamePage.puzzleWorker`
- `timerId`
- 鼠标拖拽中间态
- 撤销/重做运行时栈对象

## 8. 下一步建议

如果继续推进，建议下一次只做一个小步：

1. 新建一个真正的 `@ComponentV2` 容器，例如 `GameOverlayHostV2`
2. 先只承接 `loading/ui` 这两个 domain
3. 让 `GamePage.ets` 继续作为 V1 页面壳，给这个 V2 容器传必要参数
4. 验证 build 与行为一致后，再迁 `interaction`

这样才符合官方 MVVM V2 文档的思路：**先有 V2 View，再让 V2 View 消费 V2 ViewModel，而不是反过来让 V1 页面伪装成 V2。**

## 9. 对本项目的落地建议

如果只用一句话总结：

- **现在别在 `GamePage.ets` 里继续“接 store”了。**

更合理的路线是：

- `GamePage.ets` 继续作为 V1 页面壳和编排层
- 新增独立的 `@ComponentV2` 视图岛
- 为这些 V2 视图岛提供真正的 V2 ViewModel
- 等外围覆盖层迁移稳定后，再考虑是否继续向棋盘核心区域推进

这才同时尊重了两份官方文档：

- 对 V1 主线，按 V1 的规则维护
- 对 V2 新增部分，按 V2 的规则实现
