# 🔍 硬编码值审计与变种数独适配清单

> 全面审计代码中所有硬编码的 9×9 标准数独假设，为变种数独适配做准备。
>
> 目标：将所有硬编码的 `81`、`9`、`3`、`[1..9]` 等值替换为
> `GRID_SIZE` / `BOX_SIZE` / `TOTAL_CELLS` / `DIGITS` 常量或 `VariantConfig` 属性。

---

## 一、审计范围

### 1.1 需要替换的硬编码类型

| 硬编码值 | 含义 | 应替换为 |
|---------|------|---------|
| `81` | 总格数 (9×9) | `TOTAL_CELLS` 或 `config.grid.totalCells` |
| `9` (作为网格尺寸) | 行/列数 | `GRID_SIZE` 或 `config.grid.gridSize` |
| `9` (作为数字上限) | 最大数字 | `GRID_SIZE` 或 `config.grid.gridSize` |
| `3` (作为宫格尺寸) | 宫行/列数 | `BOX_SIZE` 或 `config.grid.boxRows`/`boxCols` |
| `[1,2,3,4,5,6,7,8,9]` | 有效数字列表 | `DIGITS` 或 `config.grid.digits` |
| `row * 9 + col` | 一维索引计算 | `row * GRID_SIZE + col` |
| `Math.floor(row / 3)` | 宫格行计算 | `Math.floor(row / BOX_SIZE)` |
| `Math.floor(col / 3)` | 宫格列计算 | `Math.floor(col / BOX_SIZE)` |
| `[0, 9, 9, 9, ...]` | 剩余计数初始化 | 动态构建，长度基于 `GRID_SIZE` |

### 1.2 不需要替换的 "9"

以下出现的 `9` 是数字值而非网格尺寸，**不需要替换**：

- 键盘映射中的数字键 `1`-`9`（当实际只有 9 个数字键时）
- Difficulty 枚举值中的数字
- 颜色值中的 `9`（如 `#FF9800`）
- 尺寸值中的 `9`（如 padding、font size）

## 四、修复策略

### 4.1 常量层（已有，确保一致使用）

`constants/GameConstants.ets` 已定义：
```typescript
export const GRID_SIZE = 9
export const BOX_SIZE = 3
export const TOTAL_CELLS = 81
export const DIGITS: number[] = [1, 2, 3, 4, 5, 6, 7, 8, 9]
```

**策略**：所有硬编码统一替换为这些常量的引用。这是最小改动方案——
常量值本身仍为 9/3/81，但后续切换变种时只需改一处。

### 4.2 剩余计数初始化

当前硬编码：
```typescript
remainingCounts: number[] = [0, 9, 9, 9, 9, 9, 9, 9, 9, 9]
```

添加工厂函数：
```typescript
// GameConstants.ets
export function createInitialRemainingCounts(): number[] {
  const counts = [0]
  for (let i = 0; i < GRID_SIZE; i++) {
    counts.push(GRID_SIZE)
  }
  return counts
}
```

### 4.3 ForEach 数组

当前硬编码：
```typescript
ForEach([1, 2, 3, 4, 5, 6, 7, 8, 9], ...)
```

统一使用：
```typescript
ForEach(DIGITS, ...)
```

宫格索引数组：
```typescript
// 当前
ForEach([0, 1, 2, 3, 4, 5, 6, 7, 8], ...)

// 修复后
ForEach(Array.from({ length: GRID_SIZE }, (_: void, i: number) => i), ...)
```

### 4.4 宫格计算统一

当前：
```typescript
Math.floor(row / 3) * 3
```

修复：
```typescript
Math.floor(row / BOX_SIZE) * BOX_SIZE
```

## 六、后续路线

本次审计完成 P0 修复后，后续变种数独适配还需要：

1. **拓扑系统切换** — 从静态 `Topology.ets`（预计算 27 units）切换到动态 `VariantTopology.ets`
2. **生成器参数化** — `SudokuGenerator` 接收 `VariantConfig` 参数
3. **解题器参数化** — `SudokuSolver` + `LogicSolver` 接收动态 units/peers
4. **UI 动态布局** — `SudokuBoard` 支持可变宫格布局（如 2×3 宫 for 6×6）
5. **笔记位宽扩展** — 12×12 需要 24 bit（仍在 32 bit `number` 内），16×16 需要 32 bit
